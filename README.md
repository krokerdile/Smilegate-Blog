# Project3 BLOG

# 22.12.25 커밋을 마무리 하면서

코드 리뷰 해주시는 멘토님들 께서 보기 편하시도록 관련된 내용을 정리해 놓은 글 부분입니다. 

react, express, mysql을 연결해서 Blog 프로젝트를 구현해보려고 하였습니다. 

## Requirements

- 글 쓰기/수정
- 글 목록/삭제
- 댓글
- UI디자인
- 관리자도구

필수 사항에서 거의 대부분을 구현 하긴 했지만, 리액트를 올해 10월 부터 제대로 공부하기 시작하고 온전히 프로젝트를 한 것이 이번이 거의 2,3번째 입니다. 거기에 기존에 사용해보지 않았던 express를 통해 백엔드를 구현하고, mysql도 사용은 해보았지만 mongodb와 같이 nosql 쪽을 좀 더 많이 사용했어서 그런지 공부하고 구현하는데 생각한 거 보다 시간이 많이 소요 되었던 것 같습니다. 

전반적인 디자인을 구현할 때도 MUI를 이번에 처음 사용해서 구현을 해보았는데, 일반적으로 프론트 쪽을 구현할 때는 어떤 식으로 가장 많이 쓰는 지 궁금하기도 합니다. 기존에 알고 있던 내용으로는 material ui와 같은 라이브러리를 통해 구현을 한 후에 자체적인 디자인을 구현해서 넘어가는 식으로 한다고 알고 있는데 프론트엔드 개발자로 취직을 생각한다면 어느 정도 수준을 요하는 지도 궁금합니다. 

최신글을 보여주는 방식은 최신글을 따로 나눠서 보여줄 수도 있지만, 최신순과 오래된 순으로 나눠서 글을 보여주는 방식으로 코드를 짜서 해결해보았습니다. 그런 방식이 좀 더 글을 보는데 괜찮은 거 같아 그렇게 구현을 해보았고 만약에 최신글을 보여주는 방식으로 해결을 한다면 짜놓은 코드에서 앞에서 10번째 글까지 따로 받아와서 보여주는 방식으로 구현을 하면 될 것 같습니다. 

저의 경우에 댓글 기능을 구현할 때 DB 에서 postId를 통해서 댓글을 참조 해오는 방식으로 구현을 해두었습니다. 이렇게 구현하는게 옳은 방식인지는 확답을 못하지만, 프로젝트를 진행할 당시에는 이 방식이 가장 효율적이지 않나 생각을 했었고, 관련된 퍼포먼스는 어떠한 부분을 고려해야 하는지에 대해 짚이는 바가 거의 없어서 혹시 만약에 이러한 기능을 구현한다면 어떤 식의 로직에서 퍼포먼스를 최대한 신경 써야 되는지도 궁금합니다. 

관리자 기능 같은 경우에는 제가 기존에는 django-rest-api를 통해서 구현을 하려고 했어서, 해당 프레임워크에서 제공하는 admin과 가장 유사하게 구현을 하려고 해보았습니다. 추가적인 관리자 기능이 마땅히 떠오르는게 없어서 최대한 필요한 것이라고 생각한 것은 다 넣고 코드를 짜 보았습니다. 

코드를 짜면서 수정이 필요할 거라고 생각한 부분들은 반응형 웹을 위해서 Responsive라는 테두리를 만들어서 덮어주는 과정들에서 코드가 많이 반복되고 불필요 했던 것 같아, 공통적인 틀을 만들때는 어떠한 방식을 사용하는 것이 가장 효율적인지 궁금합니다. 

많이 부족한 코드라고 생각을 하고 부족한 점 많이 짚어주시면 감사할 것 같습니다.

# 목표 설정

# 프로젝트 진행 하며 생긴 변경사항

💡 **Problem1 - django rest api + mysql 사용에 대한 경험 부족**

제가 기존에 Django로 개발 공부를 시작해서 django를 써서 백엔드를 구성하는 것이 효율적일 것이라고 생각을 했었으나, django restful api를 사용해서는 경험이 없어서 이게 과연 효율적 일지 처음 계획 제출 이후에 한번 더 점검을 해보았습니다. 생각보다 시간이 오래 걸릴 것이라 예상이 되는 점도 확인을 하였고 공부를 좀 더 해야 되는 부분이 커서 캠프장님께 슬랙을 통해 조언을 구하고 좀 더 추천을 받았던 express로 변경하여서 진행하기로 결정하였습니다.

</aside>

<aside>
💡 **Problem2 - 2주차에 갑작스러운 감기몸살**

필요한 자료와 레퍼런스들을 잘 찾아보고 2주차에 본격적인 시작을 하려던 와중 2주차 화요일 저녁 부터 갑작스럽게 감기몸살로 인해서 기존에 하고 있던 공부 뿐만 아니라 개인 프로젝트를 하는 것 자체가 불가능해져 버렸습니다. 몸관리 또한 필수적인 요소지만 생각지도 못하게 침대에서 나오지 못할 정도로 아프게 되면서 컴퓨터는 토요일까지 손도 되지 못하는 상황이 와버렸습니다. 기존에 계획했던 2주차 까지의 최소 완성도에 필요한 시간이 너무나도 부족해져 버리면서 컴퓨터를 사용할 수 있는 몸상태가 된 일요일에 본격적으로 개발을 시작할 수 밖에 없었습니다.

### Solution

두번째 문제로 인해서 기존계획이 완전히 틀어져 버리면서 계획을 전면 수정 할 수 밖에 없었습니다. 최소한의 기능을 구현하는 것이 우선 순위 였기 때문에 디자인 적인 부분은 뒤로 미뤄두고 2주차 일요일 하루 전체를 전반적인 구조를 짜고 개발을 하는데 몰두를 하였습니다.

react, express, mysql을 기존에 목표 했던 부분 까지 거의 만드는데 집중하여서 기능을 하는 정도 까지는 완성을 하였고, 디자인적인 부분은 어쩔 수 없이 3주차 초반부 월요일, 화요일에 채워나가야 될 것 같습니다.

</aside>
